1.1)내부링크 무결성


◎ 측정기간 동안 전체 오브젝트 수는 387개이며, 이중 결함이 있는 오브젝트의 비율은 1.03%로 결함율은 낮음.

전체 오브젝트 수 387
끊어진 오브젝트 수 4
내부링크 결함율(%) 1.03

==> 진단 의견에 문제가 있습니다.
행정기관별 보고서마다 결함율 계산 방법이 다릅니다.
기관A 내부링크 결함율=(끊어진 링크를 가진 페이지/총 HTML 페이지)
기관B 내부링크 결함율=(총 끊어진 링크/총 HTML 페이지)

==> 맥사마인 보고서에 문제가 있습니다.
끊어진 링크를 가지고 있는 페이지에 한글로 된 링크들을 모두 오류로 분류시키고 있습니다.
이건 웹사이트(utf-8)와 맥사마인 보고서(utf-8)의 인코딩 차이로 인한 것 같은데..
실제 웹사이트에서 확인해보면 이상 없음을 알 수 있습니다.
그리고 행정기관 홈페이지 구축, 운영 표준지침(2005.4) 을 보면 어디에도 utf-8 대신 utf-8로 사이트를 구축하라는 내용은 없습니다.

참조되어진 총 외부 오브젝트 수 228
연결되지 않는 외부 오브젝트 수 8

==> 진단 의견에 문제가 있습니다.
여기서는 바로 되어 있는 것 같지만, 어떤 기관의 경우..
“결함이 있는 오브젝트의 비율은 61.68%로 결함율은 매우 낮음“ 으로 표시되어 있습니다.
결함비율이 61.68% 이나 되는데, 결함율이 매우 낮다고 진단내려서 혼란을 주고 있습니다.

==> 맥사마인 보고서에는 여기에 관련된 내용이 없는데, 어떻게 진단을 하였는지 알려주십시오.
맥사마인 보고서 사이트 기본정보에 보면 "평균 (스캔) 기간: 산청군은 4.04 일" 만 나오고, 정확한 진단 기간이 없습니다.
진단 툴, 시기, 방법 등을 상세히 알아야 향후 웹사이트 성능 개선에 활용할 수 있습니다.
이번 진단 및 컨설팅의 취지는 웹사이트를 개선시키자는 것이 아닌가요?
참고로 저희가 파이어폭스-YSlow 툴로 측정해본 산청군 메인페이지 응답속도는 4초대였습니다.

◎ 상시적으로 사이트 메인페이지에 접속하였을 때 서비스의 장애 없이 가용이 되고 있는지를 점검.
* 가용성 = (성공 횟수) / (사이트 접속 횟수) * 100 (정상적인 페이지 이외 다른 페이지 내용이 표시 및 서비스의 장애가 발생, 60초 이내 완료가 안 되었을 때 가용성이 저하됨)

◎ 측정기간 동안 일정한 간격의 시간을 두고 메인페이지에 접속한 수는 192회 이며,이중 총 0건이 실패로 나타났으며 전체 가용성은 100%로 매우 높은 수준임.

2.2)응답속도


◎ 사이트의 메인페이지에 접속한 후 페이지가 정상적으로 로딩 되는지를 확인하고 완료 되는데 까지 걸리는 사용자 관점의 평균 응답시간.
* PC의 성능이나 회선의 속도의 따라 차이가 있을 수 있으므로, 현재 가장 일반적으로 사용하는 PC 사양과 ADSL 회선을 사용자 관점에서의 응답시간을 측정.
* 초기 페이지의 응답을 빠르게 하기 위해서는, 초기 페이지 구성에서 큰 용량의 이미지나 플래시 등의 오브젝트 사이즈와 수를 제한하여 전체 페이지 사이즈를 줄이는 것이 좋다.

◎ 측정기간 동안 평균 응답속도는 약 9.50초이고, 응답속도 품질분포는 전체 공공기관 사이트 중 20%에 포함되며 속도는 보통임.

◎ 사이트 메인페이지 접속 후 응답속도 대비 이미지, 플래시 등과 같은 각 요소들의 총 사이즈의 수신량을 점검.
* 시간별 수신량 =  메인 페이지 평균 사이즈 / 메인 페이지 평균 응답속도
* 수신량이 많을수록 페이지 사이즈에 대비하여 응답시간이 빠르다는 것으로 성능이 좋다는 의미이다.

2.4)콘텐츠 중복성

* 콘텐츠 중복성 = (중복 사이즈) / (전체 사이즈) * 100
* 사용자의 컴퓨터에서 사이트의 파일이나 이미지를 브라우징 시 재사용 저해. 사이트의 속도 향상을 위해 Cache화 할 때 불필요한 공간 낭비

◎ 측정기간 동안 실행된 총 데이터 크기는 104,535.48KB이며 그 중 중복된 데이터로 인해 불필요하게 사용된 공간 비율은 0.01%로 공간 낭비가 매우 낮은 편임.

3.1) 문서 타입의 준수성


◎ 문서타입이 누락된 페이지가 있는가를 점검.
* 문서타입의 준수성 = (문서타입을 가진 수) / (전체 페이지 수) * 100
* 문서타입을 정의하지 않았을 경우 브라우저나 뷰어 등에서 문제를 일으킬 수 도 있으므로, 정확한 문서타입을 정의하고 HTML 코딩을 하는 것이 좋다.

◎ 측정기간 동안 실행된 전체 페이지 중에 문서타입을 가진 페이지는 0.52%로 매우 낮은 편임.

전체 페이지수 387
문서타입을 가진 수 2
문서타입의 준수성(%) 0.52

==> 진단의견에 문제가 있습니다.
위에서 "문서타입을 가진 수" 는 맥사마인 보고서를 보니, "부정확한 Doctypes을 가진 페이지" 수 이고,
정확하게 고치면 "문서타입을 가졌지만 웹 표준 문법 오류가 발견되는 페이지"입니다.
고로, 산청군의 진단의견은 문서 타입의 결함률이 0.52%로 매우 낮은 편(매우 우수)이라고 해야 맞습니다.

그리고, 웹 표준 문법은 접근성과는 그다지 관계가 없습니다.
현재 대부분의 웹사이트 구축 시, 웹 표준 마크업 문법은 브라우저에서의 지원 정도가 차이가 있으므로, 브라우저 호환 여부만 확인하고 있습니다.
이는 개발 방향이 잘못된 것이 아니라, 웹이 흘러온 역사에 의해 그렇게 할 수 밖에 없는 구조적 한계가 있는 것입니다.

오히려 맥사마인 보고서의 "누락된 Doctypes을 가진 페이지"가 비표준 태그를 무분별하게 사용하고 있는데도 이에 대한 진단 내용은 없습니다.

==> 맥사마인 보고서에 문제가 있습니다.
"부정확한 Doctypes을 가진 페이지" 는 "문서타입을 가졌지만 웹 표준 문법 오류가 발견되는 페이지" 로 수정되어야 합니다.

※ 참고 : 웹 접근성 품질마크 인증사이트를 보면,
도움나라의 경우 올바르지 않는 문서타입인데도 접근성 품질마크를 획득하였고,
청와대의 경우 올바른 문서타입을 가졌지만 웹 표준 마크업 유효성 검사를 통과하지 못하고 있습니다.
그렇지만, 둘 다 웹 접근성이 우수한 사이트 임은 분명합니다.


3.2) 비텍스트 콘텐츠 인식성


◎ 이미지나 동영상 같은 글로 표현되지 않는 콘텐츠에 대해서는 표현하고자 했던 의미나 기능을 대체 텍스트로 설명을 해서 장애인들이 스크린 리더와 같은 프로그램으로 인식이 가능토록 한다.
*  비텍스트 콘텐츠 인식성 = (대체 텍스트를 가진 콘텐츠 수) / (전체 비텍스트 콘텐츠 수) * 100

◎ 측정기간 동안 실행된 전체 비텍스트 페이지 중에 대체 텍스트를 가진 콘텐츠 비율은 84.48%로 높은 편임.

전체 비텍스트 페이지 수 13,195
대체 텍스트를 가진 콘텐츠 수 11,147
비텍스트 콘텐츠 인식성(%) 84,48

==> 맥사마인 보고서에 문제가 있습니다.
object 에 없는 alt 속성 존재 여부를 진단하여 결함이 아닌 것을 결함이라고 표시하고 있습니다.
※ 참고 : W3C HTML 4명세 13.3 일반적 포함: OBJECT 엘레멘트


3.3) 데이터 테이블 정확성


◎ 데이터 테이블은 표로 나타내기에 적당한 논리적인 정보들을 머리글 부분과 일반 내용 부분으로 나누어 표현한 표로서 테이블을 구성하는 데이터 셀의 내용에 대한 정보가 충분히 전달될 수 있어야 한다.
* 데이터 테이블 정확성 = (데이터 테이블을 가진 페이지 수) / (테이블을 가진 페이지 전체 목록) * 100

◎ 측정기간 동안 실행된 전체 페이지 중에 테이블을 가진 페이지 전체 목록 중에 데이터 테이블을 가진 페이지 목록은 78.30%로 매우 낮은 편임.

데이터 테이블을 가진 전체 페이지 수 318
문서타입을 가진 수 249
데이터 테이블 정확성(%) 78.30

==> 진단 의견에 문제가 있습니다.
* 데이터 테이블 정확성 = (데이터 테이블을 가진 페이지 수) / (테이블을 가진 페이지 전체 목록) * 100
은 다음과 같이 수정되는 것이 맞지 않을까요?
* 데이터 테이블 정확성 = (제목 셀과 내용 셀을 구분할 수 있는 데이터 테이블 수) / (데이터 테이블 수) * 100

그리고, 표 2번째 행에서 “문서타입을 가진 수“ 라고 항목을 틀리게 표시하고 있습니다.

데이터 테이블 정확성(%)의 값이 어떻게 나온 건지 알 수 없습니다. (맥사마인 보고서로 알 수 없음)

==>맥사마인 보고서에 문제가 있습니다.
배치용 table은 제목셀과 내용셀의 구분을 할 수 없는데도 결함이 있는 걸로 진단하고 있습니다.

데이터 테이블 정확성은 수동검사로 진단하여야만 합니다.

※ 참고 : http://www.iabf.or.kr/Lab/Certification/품질마크인증심사기준.zip


3.4) 검색의 용이성


◎ 검색전용 엔진에 노출이 잘 될 수 있도록 “Title"을 가졌는지를 점검.
*  검색의 용이성 = (타이틀을 가진 페이지) / (전체 페이지) * 100

◎ 측정기간 동안 실행된 전체 페이지 중에 타이틀을 가지 페이지 비율은 94.32%로 높은 편임.

전체 페이지수 387
타이틀을 가진 페이지 수 365
검색의 용이성(%) 94.32

==> 맥사마인 보고서에 문제가 있습니다.
검색엔진 노출이 필요 없는 프로그램 처리 페이지도 오류로 검출하고 있습니다.
예를 들어 글쓰기 전송 버튼을 클릭했을 때 글을 DB에 입력한 후 처리 완료 페이지로 이동하는 그런 페이지는
서버측 내부에서 처리되는 것이기에 검색엔진에 노출되는 것이 아닐뿐더러, title 을 가질 필요도 없습니다.


3.5) 프레임 사용 준수성


◎ 콘텐츠를 구성하는 프레임의 수는 최소한으로 하며, 프레임을 사용할 경우에는 프레임별로 타이틀을 붙어야 한다.
* 프레임 사용 준수성 = (타이틀을 가진 프레임 수 ) / (전체 프레임 수) * 100

◎ 측정기간 동안 실행된 전체 프레임 수 중에 타이틀을 가진 프레임 비율은 100%로 매우 높은 편임.

전체 프레임 수 3
타이틀을 가진 프레임 수 3
프레임 사용 준수성(%) 100



3.6) 이미지맵 사용 준수성


◎ 서버 이미지맵 사용은 가급적 피하는 것이 좋으며, 클라이언트 이미지맵 사용시 longdesc 속성을 이용해 별도의 파일에서 자세하게 설명하되, 이미지에 대한 대체 텍스트를 빠뜨리지 말고 반드시 넣어야 한다.
*  이미지맵 준수성 = (대체 텍스트를 가진 이미지맵 수 ) / (이미지맵을 가진 페이지 수) * 100

◎ 측정기간 동안 이미지맵을 가진 페이지 중 대체 텍스트를 가진 이미지맵  비율은 82.04%로 매우 높은 편임.

이미지맵을 가진 페이지 수 1,136
대체 텍스트를 가진 이미지맵 수 932
이미지맵 준수성(%) 82.04

==> 진단 의견에 문제가 있습니다.
"대체 텍스트를 가진 이미지맵 수"가 아닌 "longdesc를 가진 이미지맵 수" 로 이미지맵 준수성을 계산하고 있습니다.

==> 맥사마인 보고서에 문제가 있습니다.
img 와 area 모두 alt(대체텍스트)가 있어도 longdesc가 없으면 오류로 검출됩니다.
이는 WCAG 에 “클라이언트측 이미지 맵에 있는 링크에 대한 대체 텍스트를 찾아 표시해 주는 웹 표시 장치가 널리 보급되기 전에는” 이라는 전제를 무시하고 진단한 겁니다.
현재 이미지맵은 널리 보급되어 있고, longdesc는 오히려 거의 지원되지 않습니다.
longdesc는 alt 로 부족한 경우만 사용하면 되는 것이고, 현재 오류로 검출되는 것들은 대부분 longdesc가 불필요한데도 오류로 진단하고 있습니다.

이 부분의 정확한 진단은 수동 검사로만 할 수 있습니다.

※ 참고 : WCAG 지침1 | 웹 접근성 향상을 위한 국가표준 기술 가이드라인 - 4페이지 참조
