1-3. 프로그램 Naming Rule

HTML
  • 화면명 : U+업무코드+구분번호(3 or 5).html
  • 적용예 : UHPKOR01004_02_01_01.html
  • 표기법 : 영문 대문자로만 표기
JSP
  • 소스파일명 : J+업무코드+구분번호(3 or 5).jsp
  • 적용예 : JHPKOR01004_02_01_01.jsp
  • 표기법 : 영문 대문자로만 표기
Script
  • 스크립트명 : 기능명(의미있는이름)+.js
  • 적용예 : common.js
  • 표기법 : 영문 소문자로만 표기
XML
  • 스크립트명 : 의미있는이름+.xml
  • 적용예 : user_files.xml
  • 표기법 : 전체 영어 소문자로 표기, 복합어일 경우 단어 사이를 "_"로 구분
  • BLD는 JSP명과 파일명(구분코드 J 제외)을 동일하게 하고 확장자를 달리하여 소문자로 표기.

참고 자료

4-2. 웹 접근성 기술 가이드 라인 18개 지표 및 내용

(1) 이미지의 의미나 목적을 이해할 수 있도록 적절한 대체 텍스트를 제공해야 한다.

※ 의미가 있는 이미지의 경우 대체 텍스트를 의미나 기능이 동일하게 제공
※ 의미가 없는 이미지의 경우 대체 텍스트를 blank(alt="")로 제공

  • 대체텍스트에 부연 설명이 필요하면 title 속성을, 구조적 태그가 필요하면 longdesc 속성을 이용할 수 있다.
  • 점검 : KADO-WAH 2.0 자동 평가 + 수동 점검 (소스 확인) 필요
(2) 배경 이미지가 의미를 갖는 경우, 배경 이미지의 의미를 이해할 수 있도록 대체 콘텐츠를 제공해야 한다.
  • 이미지 대체(Image Replacement) 기법을 사용하여 동등한 정보를 제공할 경우에는 준수한 것으로 판단
  • 점검 : CSS 제거 후 소스 확인 (파이어폭스 [보기]-[문서 스타일]-[스타일 제거] 등)
(3) 동영상, 음성 등 멀티미디어 콘텐츠를 이해할 수 있도록 대체 수단(자막, 원고 또는 수화)을 제공해야 한다.

* 실시간 방송의 경우는 평가에서 예외로 함

  • 요약 정보만 제공하는 경우는 부적합
(4) 색상을 배제하여도 원하는 내용을 전달할 수 있도록, 색상 이외에도 명암이나 패턴 등으로 콘텐츠 구분이 가능해야 한다.
  • 점검 : 흑백 프린터 및 모니터로 출력할 경우 콘텐츠를 구분하지 못하면 위반
(5) 서버측 이미지 맵을 제공할 경우, 해당 내용 및 기능을 사용할 수 있는 대체 콘텐츠를 제공해야 한다.

* 지리정보시스템(GIS)은 평가에서 예외로 한다.

  • 클라이언트측 이미지 맵(Client-side Image Map)의 경우 영역별로 태체 텍스트를 적절하게 제공해야 함.
  • 서버측 이미지 맵에 사용된 링크 목록이 키보드로 접근 가능한 대체 링크로 제공된 경우 접근성이 있다고 판단함.
(6) 프레임을 제공할 경우, 프레임의 내용을 이해할 수 있도록 적절한 제목(title 속성)을 제공해야 한다.
  • 점검 : KADO-WAH 2.0 자동 평가 + 수동 점검 (소스에서 frame 과 iframe 요소를 찾아서 확인) 필요
(7) 깜빡이는 콘텐츠를 제공할 경우, 사전에 경고하고 깜빡임을 회피할 수 있는 수단을 제공해야 한다.

※ 깜박임 기준: 초당 3~49번을 깜박이는 콘텐츠

  • 메인 페이지에서 깜빡임이 있는 경우에는 사전에 경고할 수 없으므로 준수하지 못한 것으로 파악
  • 동영상/멀티미디어 콘텐츠를 정지된 상태에서 제공
  • 점검 : 깜박이는 콘텐츠 여부를 자동평가도구(Flickering and photosensitive Epilepsy)를 활용하여 판단 (움직이는 GIF만 가능)
(8) 모든 기능을 키보드로 이용할 수 있어야 한다.
  • 점검 : 실제 키보드를 이용하여 앞·뒤로 탭 이동이 가능하고 웹 페이지 내용을 모두 볼 수 있으며, 서식 이용이 가능하고 기능이 제대로 작동하는지 확인
  • 링크 이미지의 테두리를 숨기기 위해 onfocus="this.blur()" 를 사용한 경우 키보드 포커스가 되지 않으므로 준수하지 못한 것으로 판단
(9) 반복되는 링크를 건너뛸 수 있도록 건너뛰기 링크(skip navigation)를 제공해야 한다.
  • 점검 : CSS 제거 또는 (X)HTML 코드로 판단
(10) 시간 제한이 있는 콘텐츠를 제공할 경우, 시간 제어 기능을 제공해야 한다.

※ 예외 : 경매, 실시간 게임 등과 같이 시간제한이 필수적인 콘텐츠

  • 점검 : 자동 갱신 콘텐츠는 사용자가 이를 제어하고 (지나간)콘텐츠를 확인할 수 있는 수단을 제공하면 준수한 것으로 판단
  • 포커스시 멈추고 이전, 다음 콘텐츠를 보는 기능이 있으면 된다.
(11) 새 창(팝업창 포함)을 제공할 경우, 사용자에게 사전에 알려야 한다.
  • 모든 팝업창은 사용자의 입력으로 발생하여야 하며, title="새 창" 등을 사용하여 사용자에게 알린다.
  • 점검 : 메인 페이지 등에 접속 시 자동으로 뜨는 팝업창이 있는지 확인하고 새 창이 뜨는 링크일 경우 소스 코드 확인
(12) 데이터 테이블을 제공할 경우, 테이블의 내용을 이해할 수 있는 정보(제목, 요약정보 등)를 제공해야 한다.
  • 데이터 테이블의 내용을 이해할 수 있도록 제목(summary 속성, <caption> 태그, <h1>~<h6> 태그)중 하나 이상 제공한다.
(13) 데이터 테이블을 제공할 경우, 제목 셀과 내용 셀을 구분할 수 있어야 한다.
  • 제목 셀(<th>)과 내용 셀(<td>)을 구분할 수 있는 태그 제공
  • <thead>, <tbody>, <tfoot> 등 행단위 구분을 위한 태그 제공
  • 모든 헤더 셀에 scope="col" 또는 scope="row" 속성을 사용하여 해당 행 또는 열의 헤더 구분을 권장
(14) 해당 페이지를 잘 이해할 수 있도록 페이지 제목(<title>)을 제공해야 한다.
  • <title>이 페이지를 구분할 수 있게 간단 명료하게 표현한다.
  • 페이지 제목을 자바스크립트로 구현하거나, URL을 고정하기 위해 의미없는 프레임을 사용한 경우는 부적합
(15) 콘텐츠는 논리적인 순서로 구성되어야 한다.
  • 점검 : 콘텐츠 배치 순서, Tab 이동 순서, 서식(form)의 이동 순서 확인
(16) 온라인 서식을 제공할 경우, 레이블(<label>)을 제공해야 한다.
  • 필드를 여러 개로 나누어 입력하는 경우 입력란을 구분할 수 있도록 title 속성을 제공
(17) 애플릿, 플러그인(ActiveX, 플래시) 등 부가 애플리케이션이 제공하는 경우, 해당 애플리케이션이 자체적인 접근성을 준수하거나 또는 사용자가 대체 콘텐츠를 선택하여 이용할 수 있어야 한다.
  • 부가 애플리케이션을 제공할 경우에는 해당 애플리케이션을 최신 보조기술 수준에서 사용할 수 있도록 a)자체적인 접근성, b)대체콘텐츠 제공 중 한 개를 충족하여 제공해야 한다.
    ※ 예외사항 : 공인인증서, 보안결재의 경우에는 접근성을 보장하는 애플리케이션이 2008년 12월말 기준으로 국내에 존재하지 않으므로 예외로 처리(단, 접근성 보장 애플리케이션이 등장할 경우까지만 예외로 인정)
  • 플래시로 구현된 내비게이션의 메뉴 이동 순서가 올바르지 않으면 자체적 접근성 미준수로 판단
    ※ 플래시로 구현한 내비게이션은 플래시 기술을 제작한 Adobe에서 제공하는 접근할 수 있는 플래시(Best Practices for Accessiblel Flash Design : http://www.adobe.com/resources/accessibility/best_practices/best_practices_acc_flash.pdf) 방법을 이용해서, 대체 텍스트와 이동 순서를 보장해야 함.
  • 점검 : ActiveX의 보안프로그램 등의 애플리케이션 이용시 키보드 이용보장, 서식에 대한 레이블 제공, 이미지에 대한 대체 정보 제공 등 자체적인 접근성 지침의 준수 여부를 확인
  • 플러그인을 사용하지 않을 때, 기능을 사용할 수 있는 대체 콘텐츠가 존재하면 준수한 것으로 판단
  • 점검 : 오페라 - [도구]-[빠른 설정]-[플러그인 사용] 체크 해제.
  • 사이트 전체를 플래시로 만들었지만 동등한 내용의 접근할 수 있는 대체 콘텐츠를 제공하고 있으면 준수한 것으로 판단
(18) 마크업 언어로 구현할 수 있는 기능(링크, 서식, 버튼, 페이지 제목)을 자바스크립트로만 구현하지 말아야 한다.
  • 링크의 값을 올바르게 제공하지 않고 자바스크립트로 제공한 경우 준수하지 못한 것으로 판단
  • 점검 : 파이어폭스의 부가기능 Web Developer 설치 후 Web Developer 도구모음에서 [차단]-[JavaScript 사용안함]-[모든 JavaScript] 체크

개정된 지침에서는 "마크업 언어로 구현할 수 있는 기능(링크, 서식, 버튼, 페이지 제목)을 자바스크립트로만 구현하지 말아야 한다." 라는 내용이 제외되었는데..
2010.03에 발표한 웹 접근성 품질마크 심사 기준을 예의 주시하자!

(1) (명확한 지시사항 제공) 지시사항은 모양, 크기, 위치, 방향, 색, 소리 등에 관계없이 인식될 수 있어야 한다.
  • “윈도우를 닫으려면 왼쪽 상단에 있는 동그란 빨간 버튼을 클릭” 한다는 지시문의 경우 구체적인 설명이 부족하여 이를 이용하지 못할 우려가 있음.
  • “왼쪽 상단의 첫 번째 동그란 빨간 버튼인 창 닫기 버튼을 선택하시면, 윈도를 종료하실 수 있습니다.” 처럼 지시사항에 대한 명시적인 설명을 제공하여 해결할 수 있다.

(10) 지역 초기화를 위한 CSS 코드 참조
  • 공통 선택자 * 를 이용하여 margin, padding 등이 전역적으로 초기화 선언되지 않은 문서에 새 스타일을 안전하게 추가하는 방법을 설명하고 있습니다. 공통선택자를 이용하여 margin, padding등이 초기화되지 않은 상태로 작성된 오랜 문서에 뒤늦게 공통선택자를 이용하여 전역 초기화를 시도하게 되면 기존의 화면 그리드에 변화가 생기므로 새 스타일이 추가되는 지역에 한하여 부분적으로만 초기화를 시도하여야 합니다. 아래 참조 코드는 공통 선택자를 부분적으로만 초기화시켜서 기존의 스타일에 영향을 미치지 않도록 안전하게 초기화 코드를 작성하는 방법입니다. .attach * { margin: 0; padding: 0; font-family: dotum; }
(11) 미디어 타입별 CSS 코드의 작성
  • 각종 미디어(프린터, 모바일, 보조공학기기 등)에 대응하는 별도의 CSS 코드를 작성할 수 있습니다. 별도의 미디어에 대응하는 CSS 문서를 따로 생성한뒤 link 또는 import 요소에 미디어 타입을 지정할 수 있으나 이렇게 미디어 타입별로 CSS 파일을 분리하는 것보다는 하나의 파일에 여러가지 미디어 대응 코드를 한꺼번에 모두 작성하는 것이 좋습니다. 한편 W3C 표준에 의하면 CSS가 다양한 미디어에 대응이 가능하도록 스펙이 정의되어 있지만 현실적으로 이것을 가장 안전하게 사용할 수 있는 경우는 인쇄장치 뿐입니다.
  • 아래는 W3C의 CSS 2.0 에 명시되어 있는 미디어 타입의 종류 입니다. all (모든 출력 장치)
    aural (음성 출력)
    braille (점자 출력)
    handheld (포켓, 모바일)
    print (인쇄)
    projection (투사 장치)
    screen (스크린, 모니터)
    tty (전신 타자기, Tele Type Writer)
    tv (텔레비전, Television)
  • 인쇄장치에 대응하기 위한 별도의 CSS 코드를 작성하려면 default.css 문서의 끝부분에 다음과 같은 형식으로 기술합니다. @media print {
    #head,
    #side,
    #foot { display: none !important; }
    }
(12) CSS 주석 가이드
  • CSS코드는 의미있는 형태로 그룹핑 하고 필요한 경우 그룹핑 된 코드 더미 첫 줄에만 주석으로 설명합니다.
  • 주석에 사용되는 언어는 한글이든 영문이든 무방하며 가능한 직관적인 단어를 선택합니다.
  • 주석과 관련된 버그를 피하기 위하여 주석 기호와 코멘트 사이에는 빈 공간을 두어야 합니다. /*head*/ (X)
    /* head */ (O)
  • 주석 안쪽에는 코드를 설명하는 문장 외에 다른 기호등을 넣지 않습니다. /** head **/ (X)
    /*- head -*/ (X)
    /*--- head ---*/ (X)
    /* head */ (O)
  • 단, CSS코드의 최상위 분류인 Tag, Layout, Class,.. 는 ========== 기호를 사용하여 시각적으로 분리하는 것을 허용합니다. /* Layout ========== */
(13) CSS 코드의 정렬
  • 선택자 기준으로 개행하고, 속성 단위로는 개행하지 않습니다. 들여쓰기를 하지 않습니다. (O)
    #selector1 { property: value; property2: value; }
    #selector2 { property: value; }
    #selector3,
    #selector4,
    #selector5 { property: value; }

    (X) 속성 단위로 개행하지 않습니다
    #selector1 {
    property: value;
    }
    #selector2 {
    property: value;
    }

    (X) 선택자 단위로 개행되어야 합니다
    #selector1, #selector2, #selector3 {
    property: value;
    }

    (X) 들여쓰기를 하지 않습니다
    #selector1 { property: value; }
    #selector2 { property: value; }
    #selector3 { property: value; }
  • 단, 화면 배치의 근간을 이루는 #wrap, #side, #body 의 종속 관계는 들여쓰기를 허용합니다.
  • CSS 코드의 정돈을 위하여 코드의 그룹간 행(line) 간격은 1행 이상 비우지 않습니다.
  • 그룹 안에서 의미있는 객체를 구분하기 위하여 1행씩 빈 행을 만드는 것은 허용 됩니다. (O)
    /* 그룹1 */
    #selector1 { property: value; }
    #selector2 { property: value; }

    /* 그룹2 */
    #selector3 { property: value; }
    #selector4 { property: value; }

    (X)
    /* 그룹1 */
    #selector1 { property: value; }
    #selector2 { property: value; }


    /* 그룹2 */
    #selector3 { property: value; }
    #selector4 { property: value; }
(14) 브라우저 Hack의 사용
  • CSS Hack 은 Hack 의 도움 없이는 불가능할 경우 최후의 수단으로만 사용합니다.
  • W3C CSS Validation Pass를 체크하여 오류가 발생하는 CSS Hack 은 사용하지 않습니다.
  • 다음은 문법적으로 유효한 CSS Hack 입니다. * html #selector { property: value; }/* IE4~6, Mac IE4~5 */
    *:first-child+html #selector { property: value; }/* Only IE7 */
(15) z-index 적용 가이드
  • z-index의 증감이 필요한 경우 가능하다면 10 단위로 증감하고 10을 초과하지 않는 범위에서 지정하는 것을 권장합니다.
  • 10 단위로 z-index를 증감하는 것은 또 다른 z-index값의 출현에 유연하게 대처하기 위함입니다.
  • 예외상황이 발생하는 경우 반드시 가이드에 따를 필요는 없습니다. (X) 두 번째 선택자에 필요 이상의 값이 들어감
    #selector1 { z-index: 10; }
    #selector2 { z-index: 100; }

    (O) 필요한 값만큼만 증가됨
    #selector1 { z-index: 10; }
    #selector2 { z-index: 20; }
