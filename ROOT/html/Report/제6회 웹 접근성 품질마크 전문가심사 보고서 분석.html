제6회 웹 접근성 품질마크 전문가심사 보고서 분석

v2009.10.08. 제6회 웹 접근성 품질마크 전문가심사 보고서 2009.9 - 한국정보화진흥원 - 진해시, 창녕군 보고서 참조하여 작성

최근 웹 접근성 품질마크 심사 동향을 고려한 지표[18]별 분석

  1. 이미지 대체텍스트 alt
    * 평가의견: 대체텍스트의 내용이 이미지의 내용에 비하여 부족한 경우가 있음.
    - 위치: 메인 이미지 메뉴명, 팝업존(이미지홍보), 도표, 찾아오시는 길, 위치지도, 시가 가사 등..
    * 준수예시: 사이트의 제목이 되는 로그 이미지의 대체텍스트로 "○○시 로고" 라고 할 필요 없다. 그냥 alt="○○시" 면 충분하다.
  2. 배경이미지가 의미를 갖는 경우
    이미지 대체 기법을 사용하여 동등한 정보를 제공하면 문제없다.
  3. 동영상, 음성
    동일한 내용의 원고라면, 동영상 페이지에서 바로 보여주거나 원고 txt파일 링크만 제공하여도 된다.
  4. 색상 이외에도 명함이나 패턴으로 구분
    * 평가의견: 색상을 배제하면 정보 전달에 문제 발생 부분.
    - 사례1: 시군 위치도 플래시 범례, 찾아오시는 길 플래시(지도에서 색으로만 길을 표현하고 있음), 설문조사 결과 도표 이미지가 지적되어 부적합.
    - 사례2: 찾아오시는 길의 고속도로 구분(범례)이 색에 의존되어 있음 ==> 1개만 지적되어 적합 판정.
    오시는 길 플래시가 빈번하게 지적되고 있으므로 운영중인거는 지적 부분 수정하고, 개편시에는 콘텐츠 제공 방식을 수정할 필요가 있다.
  5. 서버측 이미지맵 대체콘텐츠는 사용하지 않으니 지적된 적이 없다.
  6. 프레임 제목(title속성)
    프로그램용 빈 프레임에도 <iframe title="빈 프레임"></iframe> 처럼 제공하자.
  7. 깜빡임
    marquee, blink 태그와 움직이는GIF를 사용하지 않으니 지적되지 않았다.
    * 준수예시: 웹 사이트는 정지된 상태의 동영상 제공하고, 사용자가 재생 여부를 판단해야 함. 어떠한 동영상도 자동 재생되지 않아야 함.
  8. 키보드 운용
    * 평가방법: 키보드의 탭, 시프트 탭, 방향키, 엔터 등을 이용하여 화면에 보이는 요소를 제어할 수 있는지 정성평가
    * 평가의견: 키보드를 이용하여 순서대로 접근할 때 논리적이지 못하다고 부적합 판정 사례 있음.
    - 위치: 본청 메인 (직원 찾기 입력서식과 검색버튼) ==> 불필요한 tabindex="0" 로 인해 순서대로 접근할 수 없었음.
    - 위치: 문화관광 메인 (공지사항, 관광Q&A 탭메뉴)
    ==>(주) 이의제기! 논리적 순서는 지표15에 있는데~ 여기서 중복 심사하여 부적합 판정하는 것은 심사오류 라고 생각함.
    또한, 탭콘텐츠의 경우 웹접근성 연구소 와 흡사한 방식으로 접근이 가능한데도,
    탭→콘텐츠→탭→콘텐츠 순서가 맞는것 처럼 잘못된 교육 및 평가하는 전문가(?)가 간혹 있으니 주의바람.
    • 반드시 올바른 순서대로 마크업 코딩을 하고 tabindex는 사용하지 말자!
    • copy&paste 로 tabindex="0" 가 이유 없이 남아 있는 웹페이지는 탭이 올바르게 가는지 확인을 하고 tabindex="0" 는 삭제하자!
    • 모든 앵커와 서식에 tabindex="0" 를 사용한 사이트는 현재 논리적 순서에 이상없으면 그대로 두고, 개편시에는 모두 삭제하도록 하자.
  9. 스킵 내비게이션
    메인, 서브메인에서 불필요한 "부메뉴 바로가기" 링크가 있고, 검색결과 페이지에 스킵 내비게이션이 제공되지 않아서 부적합 판정 사례 있음.
    불필요한 링크는 제거하고, 본문 내용 위에 메뉴가 있는 웹페이지에는 스킵 내비게이션을 제공하자.
    * 준수예시:
    • 1) 화면에서 감춰진 스킵 내비게이션은 지체장애 등에는 도움이 되지 못함.(그래도 적합 판정)
    • 2) 키보드로 접근했을 때만, 스킵 내비게이션이 화면에 보이게 하는 방법도 있다.
    • 3) 항상 화면에 보이도록 하는 방법은 모든 사람에게 도움이 될 수 있어 더욱 바람직함.
    위의 1) 형태만 사용했는데 향후 3) 형태로 개선하는 것을 고려하자
  10. 시간 제어 기능
    * 평가의견: 배너 자동 롤링 등.. 키보드 포커스시 정지하거나, 별도의 버튼으로 정지할 수 있도록 함이 권장 됨.
    • mticker.js [initmTicker(); 2008.04.11버전] 사용시 적합 판정을 받기는 하지만,
      자동 롤링을 멈추는 수단이 마우스에 의존하고 있어 2008년 웹 접근성 실태조사에서 준수율 90~95% 정도만 받았다.
      mticker.js [initmTicker(); 2009.09.14버전] mticker.js [initmTicker(); 2009.10.13버전] 으로 업글하자! (js 파일만 교체하면됨)
      키보드 포커스시 정지가 가능하고, 정지 및 시작 기능을 제공할 수도 있다.
    * 평가의견: 검색결과의 인기검색어 롤링이 시간 제어 기능이 없어서 지적됨.
    유용하지 않는 기능이므로 가능하면 삭제하자.
  11. 새창(팝업창)
    * 준수예시:
    • 사용자가 버튼이나 링크 등을 선택한 경우에만 팝업을 활성화시킴
      • 팝업창은 링크(a태그)의 title 속성을 이용하여 title="새창으로 열림" 과 같이 미리 경고를 주어야 한다.
      • 단, target="_blank" 을 이용하여 팝업을 생성하는 경우는, 팝업을 알리는 명시적인 코드가 소스 안에 존재하므로 새창을 알리지 않아도 감점하지 않음.
        (센스리더 최신 버전에서 a요소 target을 _blank로 설정하면 '새 창'이라고 음성 출력한다.)
    • 최근 팝업창을 웹 페이지 내에서 레이어 형식으로 처리하는 경향이 있는데, 이는 콘텐츠 선형화(논리적 구성)에서 점검하고 있음
      ==>(주) 앞으로는 사전 경고 없이 다른 콘텐츠를 가리는 (레이어)팝업은 모두 감점 대상이 된다.
    • 팝업을 사용하는 경우에는 대체 텍스트를 제공하는 등 보통의 페이지와 동등하게 웹 접급성을 고려할 것.
  12. 데이터테이블 summary, caption
    * 평가의견:
    • 게시판 테이블에서 summary="게시물을 볼 수 있고 제목 링크를 통해서 게시물 상세 내용으로 이동합니다." 라고 동일하게 제공되는 것보다는 해당 게시물의 내용과 관련된 요약 정보를 제공하는 것이 바람직함.
    • 데이터테이블에 summary나 caption 등을 제공하고 있음.(적합판정)
      하지만, 현재보다 좀더 편리하게 이용할 수 있도록 현실적인 정보를 제공해야함.
  13. 데이터테이블 thead, tbody, th
    * 평가의견: 달력 테이블의 요일에는 th로 제목임을 표기함이 옳음.
    ==> 달력 테이블은 데이터테이블이므로 요일은 thead 안에 th를, 날짜는 tbody 안에 td 로 표기하자!
  14. 웹 페이지 제목 title 태그
    * 평가의견: 모든 title 태그에 공통적으로 '○○시 - 1차메뉴 - 2차메뉴'로 제공됨은 반복적인 부분이므로 가급적 생략함이 권장됨.
    ==>(주) 반대한다!! 해당 웹페이지의 구체적 정보를 제공하고 메뉴 경로까지 알려주는데 뭐가 문제인지??
    * 준수예시 :
    • 각 웹 페이지의 특성을 나타내는 고유한 제목을 가져야 함.
      ==> 현재 우리가 구축중인 웹사이트의 title 태그는 menu_all 배열에 없는 하위 메뉴명은 가져오지 못하므로 같은 title 태그를 가질 수 있다. 향후 고유한 제목을 가지게 보완이 필요하다.
    • "####" 문자와 같이 시각적인 꾸밈과 "~에 오신 것을 환영합니다" 와 같은 불필요한 요소는 사용하지 말자.
    • URL 을 고정시키기 위해 의미 없는 프레임을 사용한 경우는 부적합 판정.
      - 단, 서버측 프로그램을 이용하여 프레임 페이지의 title 태그 내용을 적절히 바꾸어 주면 준수한 것으로 인정.
  15. 논리적 순서
    * 평가의견: h태그는 문서상 순서대로 등장함이 좋으며 h1의 경우 페이지에 한번 등장함을 권장함.
    ==>(주) 의견은 옳다고 여기지만, 웹페이지는 일반적으로 내용과 메뉴가 결합되어 있으므로 권장대로 할 수 있는 구조가 아니다. 전체 웹페이지에서의 최상위 제목(웹사이트 로고) h1 과 내용에서의 최상위 제목 h1 이 필요한다고 본다.
  16. 서식 lable
    *평가의견:
    • 레이블이 제공되기 어려운 부분에는 title 속성이 적절히 제공되고 있다고 적합 판정 사례 있음.
    • 온라인 서식에 레이블이 제공되지 않아서 부적합 판정 사례 있음.
      개별주택가격 열람에서 지번/부번 입력서식, 직원 찾기 입력서식, 검색 select (민원사무편람, 구인구직, 통합검색 결과) 지적됨.
    lable 연결 안된 서식을 무조건 부적합으로 잘못 평가하는 경우도 있으므로, 심사기관에 이의를 제기하자!
    ==>(주) 작업참고: label for="" 연결, title="" , select 첫번째 oprtion 제목 방법 모두 인정
  17. 애플릿, 플러그인(ActiveX, 플래시)
    * 평가의견: 플래시가 wmode="transparent" 로 제공하고 있어서 해당요소를 인식하지 못할 수 있으므로 wmode="window" 를 권장함.
    청사안내도, 민원실 배치도, 찾아오시는 길, 위치도 지적됨.
    • 플래시는 항상 wmode="window" 로 사용하고, 안되면 "opaque" 사용하며, 그래도 안되는 경우에만 "transparent" 를 사용하자!
    • 스크린리더로 테스트 하는지는 알 수 없지만 현재까지의 행안부 웹 접근성 심사에서 볼 때
      플래시에서 키보드 탭으로 대부분의 콘텐츠 이용이 가능하고, object 대체콘텐츠가 적당히 제공되면 적합 판정을 받았었다.
      하지만, 플래시 콘텐츠의 접근성 심사 기준을 강화하는 추세이므로,
      앞으로 플래시 자체 접근성(탭인덱스와 대체텍스트)과 object 대체콘텐츠를 적절히 구성하고
      스크린리더 사용시에도 문제가 없도록 해야겠다.
  18. 자바스크립트
    * 평가의견: 링크, 서식, 버튼, 페이지 제목 등이 자바스크립트에 의존되어 제공되고 있음.
    - 위치: 검색결과(검색결과 더 보기, 검색분류, 인기검색어, 검색도움말) 페이지 지적만으로 부적합 판정 사례 있음.
