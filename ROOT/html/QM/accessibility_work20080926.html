자동 평가 대비

모든 사이트 본청 KADO-Wah 검사 100% 준수율 나오게 수정한다.
DB 내용으로 인하여 100%가 안될시 95%만은 넘도록 조치하자.
본청 하위 폴더에 존재하는 마이크로 사이트로 인하여 100% 안나오면 (마산은 약45%)

본청 메인과 접근성 준수율 낮은 항목 집중 보완

  • 본청 메인 페이지는..
    • 웹 접근성 작업 참고에 따라 모든 접근성 항목을 보완한다.
    • 유지보수로 변경된 콘텐츠.. 이미지홍보(팝업존), 팝업 창 등을 반드시 점검한다.
  • 1-1-2. 대체 텍스트의 내용은 적절하게 제공되어야 한다.
    • [김해] 날씨(플래시 대체텍스트), 역점시책(많은 내용을 담고 있는 이미지), 아이디/비번찾기(의미 있는 이미지, 의미 없는 이미지)
    • [함안] 기본현황(전문 이미지), 조직도
    • "시의회 심벌" 등을 불필요한 표현이라고 심사하는 경우도 있었으나, 향후에는 감점하지 않으므로 그대로 둔다.
    • 오시는 길 약도에 대한 설명글이 없다면 클라이언트에게 요청하고, 그래도 받을 수 없다면 약도를 보고 나름 창작해서 대체텍스트를 넣자!
  • 1-2-1. 동영상 등의 시청각 멀티미디어 콘텐츠에는 동기화된 자막이 제공되어야 한다.
    • [함안] 동영상 갤러리(원고 전문) - 문화관광 도메인이 달라도 점검함.
    • 영상 내용과 동일한 내용의 텍스트 콘텐츠를 제공하여도 된다.
    • 동영상 내용 요약만 있으면 점수 차감된다.
    • 동영상 원고 자료가 없더라도, 비동기 자막(원고)을 제공할 공간은 마련해둔다.(접근성 작업을 했다는 근거를 마련)
    • 원고를 구할 수 없으면, 해당 동영상으로 가는 링크를 주석처리하는 등의 대책을 클라이언트와 협의하자!
  • 1-3-1. 색상으로 표현된 정보는 색상을 배제하여도 원하는 내용을 전달할 수 있어야 한다.
    • [진주] 회원가입(필수입력사항)
  • 1-3-2. 전경색과 배경색은 충분한 대비를 가지고 있어야 한다.
    • [진주] 하단 영역의 이미지 텍스트, 롤오버 이미지 텍스트, 로그인 영역 배경색과 폰트색, 조직도, 회원가입 폼의 작성안내 문구 등
    • 충분한 대비? 배경색이 흰색(#FFF) 일경우 전경색은 #777 보다 짙은색이어야 한다.
    • 전체적으로 이미지 텍스트 색상..
      • 이미지 텍스트 일경우 안티알리아싱 효과로 뿌옇게 보이므로 더욱 많은 대비의 색을 적용하여야 한다.
    • 2008년까지는 이전 기준 그대로 적용하므로, 객관적 평가 방법이 나오기전에는 감점에 주의해야 한다.
      주메뉴, 좌메뉴, 본문 메뉴, 하단 정보, 사이트맵 등에서 지적된 사례가 있으므로, Fujitsu ColorSelector로 검사하여 이상이 없도록 하자.
  • 1-3-3. 텍스트는 쉽게 읽을 수 있도록 충분한 크기로 제공되어야 한다.
    • 폰트크기 11px 이하는 사용을 자제한다.
    • 2008년까지는 이전 기준 그대로 적용하므로, 객관적 평가 방법이 나오기전에는 감점에 주의해야 한다.
      하단 정보, 메뉴, 버튼(more, go 버튼 포함), 각 페이지 요약 설명 등에서 지적된 사례가 있으므로 12px이상으로 수정하고,
      디자인을 저하시킬 우려가 있는 경우, 작업량이 아주 많은 경우에는 보류하고 내역을 남겨놓자!
    • IE6 브라우저에서 텍스트크기 확대축소가 가능해야한다. body { font-size: 75%; } 만 12px로 하여도 품질마크 심사시 지적되었다.
      • 글자크기로 계속적인 민원이 문제가 된다면, 홈페이지 이용안내 혹은 날개메뉴 등에..
        http://www.mic.go.kr/ 의 우측 "글자가 작거나 크게 보이세요?" 와 같이 사용법을 넣는 방법을 고려해보십시오.
  • 2-6-2. 팝업창을 불필요하게 사용하지 않아야 한다.
    • [김해] 회원가입, 아이디/비번찾기 (팝업창 사전경고, 대체 텍스트)
    • [함안] 회원가입, 아이디 중복확인 (팝업창 사전경고, 대체 텍스트)
    • [진주] 민원안내, 시장인사말 동영상보기 버튼, 메인페이지 로딩시 (팝업창 사전경고, 대체 텍스트)
  • 3-1. 데이터 테이블의 구성 * 프로그램

      (예) summary="목록" 처럼 부실한 요약 설명을 아래 예들을 참고하여 수정한다.

      summary="공지사항 게시물 목록을 볼 수 있고 제목 링크를 통해서 게시물 상세 내용으로 이동합니다."
      summary="각 게시물의 번호, 제목, 파일, 작성자, 작성일, 조회를 볼 수 있고 제목 링크를 통해서 게시물 상세 내용으로 이동합니다."
  • 3-2-1. 의미에 맞는 적절한 HTML태그를 사용하여야 한다.
    • [함안] 전체 프레임 사용이 문제(항상 동일한 제목이 제목표시줄에 표시됨)
  • 3-2-3. 레이아웃 테이블 작성에 테이블의 구조 정보를 정의하는 태그를 사용하지 않아야 한다.
    • [진주] 교통 과태료 (레이아웃용 중첩 table 에 tbody 를 사용해서 지적됨. thead, tfoot, caption, th 등도 사용하면 안됨)
  • 3-3-1. 각 서식 제어 요소를 설명하는 label을 사용하여야 한다. * 프로그램
    • [김해] 회원가입
    • [진주] 로그인, 실명인증, 글쓰기, 검색, 회원가입(체크박스 등)
    • 정보수정, 아이디 찾기, 비밀번호 찾기, 우편번호 찾기 등 회원 로그인 관련 모든 페이지 꼼꼼하게 체크해서 수정해야 합니다.
    • 텍스트 입력상자 중에 label 을 붙일 곳이 없는 경우(예: 아이디 글자를 입력란안에 배경 처리)에도 <label for="id" class="dpn">아이디</label> 를 넣어주세요.
      ※ 여기서 class="dpn" 은 스타일시트에 .dpn { display: none !important; } 로 정의해 놓았음.
      없는 경우 같은 기능의 class 를 사용하거나 css 파일에 만들어 놓으시길 바랍니다.
      인라인스타일로 직접 display: none; 하면 스크린리더에서 읽지 않으므로 이렇게 class 로 지정해야 합니다.
  • 3-3-2. 키보드만으로도 온라인 서식 입력 및 서식 간 이동이 가능하여야 한다. * 프로그램
    • [진주] 실명확인(주민번호 앞자리 입력후 뒷자리로 자동 포커스되면 안됨)
    • 전진 이동 뿐만 아니라 후진 이동(입력 후 그 전 서식을 수정)도 반드시 가능하도록 한다.
  • 4-1-1. 스크립트, 애플릿, 플러그인 등 부가 어플리케이션에 의한 콘텐츠는 자체적인 접근성을 지니고 있어야 한다.
    • [진주] 로그인, 회원가입, ID찾기, 비밀번호 찾기를 하기위해 Active-X를 설치해야만 하는 문제
    • flash 콘텐츠가 순서에 맞게 키보드로 이동할 수 있어야 한다.
      ※ 플래시로 구현한 내비게이션은 플래시 기술을 제작한 Adobe에서 제공하는 접근할 수 있는 플래시(Best Practices for Accessiblel Flash Design) 방법을 이용해서, 대체 텍스트와 이동 순서를 보장해야 함.
    • 제3회 웹 접근성 품질마크 전문가심사 평가보고서 부산광역시 경우를 보면..
      "웹 사이트에서 제공하는 동영상이 MS Windows에서만 이용할 수 있는 wma 파일로 제공하고 있음" 이라는 심사의견이 있으나, 다른 부분에서의 지적 사항이 없어서 준수율 90%로 적합 판정을 받음.
  • 4-1-2. 부가 애플리케이션이 작동하지 않을 경우에도 동일한 내용을 전달할 수 있는 대체 콘텐츠를 제공하여야 한다. * 프로그램
    • [김해] 토론방(뷰 링크를 자바스크립트에 의존적으로 구현)
    • [진주] 메인 탭안의 리스트 링크, 실명인증 확인 버튼, 게시판 입력 후 등록확인 버튼 등 (javascript 에 의존한 콘텐츠 링크 및 서비스)
    • 플래시, 동영상 등 object 요소를 페이지에 직접 코딩(javascript에 의존하지 않도록)하고 대체콘텐츠를 보완한다.
    • javascript 없이도 서식 이용이 가능하도록 수정한다.
