== 전달사항 ==

1. PM(기획)은 프로젝트의 부속사이트(하위도메인 또는 하위폴더) 여부와 과업범위를 먼저 파악해 둡시다.
  • 영업 단계(계약전 사이트 분석)에서 파악하는 것이 바람직하지만, 안된다면.. 기획(설계)단계에서 반드시 정리합시다.
  • 부속사이트로는 로그인, 회원, 커뮤니티, 열린시장실, 시민참여시스템, 문화관광, 인터넷방송, UCC, e-Book, 팝업으로 보여주는 웹페이지 등이 모두 포함됩니다.
  • 1일을 꼬박 투자하더라도 정리해두는 것이 나중을 위해서라도 좋을 것입니다.
  • 정리가 되면 프로젝트 참여자 모두에게 전달하고 대책을 협의합시다.
  • 과업범위가 아닌 콘텐츠로 인하여 평가에 영향을 줄 수 있다는 것을 클라이언트에게 반드시 알립시다.
  • 클라이언트에게 접근성 작업을 위한 대체텍스트 자료, 도메인 정책 등을 사이트 개발 전에 요청합시다.
  • 과업의 최우선 목표(예: 품질마크 획득)가 있다면 과업 수행시 그것에 집중할 수 있도록 계속 유도합시다.
2. 모든 페이지 작업 후 접근성 준수여부를 기록합시다. (작성자: 전담 콘텐츠디자인코더, 프로그래머 각자)
  • 메뉴 엑셀 파일을 토대로 항목을 추가하여 작성합니다.
  • 메뉴 또는 메인에 링크된 부속사이트, 부속/새창 웹페이지 등도 모두 포함하여 정리합니다.
  • 적합/부적합을 표시하고 부적합인 경우 사유와 대책까지 적어둡시다.
  • 내부완료와 동시에 접근성 작업 여부 정리를 완료합시다.
  • 이후 보완 작업 후 계속적으로 갱신합니다. (→ 론칭 → 각종 평가 → 지속적으로 보존하면서 관리)
  • 언제든 전체 과업의 접근성 작업 정도를 파악할 수 있도록 보존합니다. (퇴사시 인수인계 문서에도 포함)
3. 모든 웹 접근성 작업은 1번 할 때 제대로 하는게 중요합니다.
  • 사이트 준공 후 웹 접근성 작업을 한다는 생각은 절대 하지 마십시요.
  • 론칭 전에는 수정 처리로 바쁘고, 이후는 다른 프로젝트에 투입되므로 시간이 없습니다.
  • 나중에 다시 하게 되면 작업 투입 시간은 전체적으로 손실입니다.
  • 이로 인해 과업완료에 차질이 예상되면 추가 인원 지원을 강력히 요청하십시요.
  • 구현 방법을 모르거나 혼동될 경우 코딩팀에게 도움을 요청하시면 됩니다.
